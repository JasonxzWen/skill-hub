## ADDED Requirements

### Requirement: Report structure supports scan-first navigation
The system SHALL organize generated interaction artifacts with a scan-first information architecture that separates conclusion, grouped navigation, rich content, evidence, verification, and next actions.

#### Scenario: Generate grouped report navigation
- **WHEN** the artifact input contains multiple sections across summary, diagram, code, evidence, verification, or action categories
- **THEN** the generated artifact groups navigation entries by section purpose rather than rendering every section title in one flat list
- **AND** the navigation exposes each section's title and status when provided
- **AND** the navigation remains usable when at least twenty sections are present

#### Scenario: Desktop navigation does not cover content
- **WHEN** the report is viewed in a desktop-width browser viewport
- **THEN** grouped navigation remains visually separate from the main reading column
- **AND** navigation does not overlap the hero, section headers, diagrams, code blocks, evidence cards, or action controls

#### Scenario: Narrow navigation remains readable
- **WHEN** the report is viewed in a narrow browser viewport
- **THEN** grouped navigation collapses, wraps, or scrolls without causing body-level horizontal overflow
- **AND** long section titles remain discoverable through visible text, wrapping, truncation with title text, or an equivalent accessible affordance

#### Scenario: Report keeps conclusion first
- **WHEN** a report is generated from valid input
- **THEN** the first reading area after page metadata presents the report title, status, summary, and top-level outcome before detailed sections
- **AND** evidence, verification, and next actions remain reachable from both the grouped navigation and the page flow

### Requirement: Reports meet a visual quality contract
The system SHALL apply consistent visual quality rules for typography, spacing, contrast, rich content containers, and responsive layout in generated interaction artifacts.

#### Scenario: Typography remains readable
- **WHEN** a generated artifact contains body text, metadata, card text, headings, code, and table content
- **THEN** the report uses explicit font size, line height, and font weight rules for each content class
- **AND** body and code text are not rendered with a visually fragile ultra-light weight
- **AND** line height is tight enough for dense engineering reading without causing text collision

#### Scenario: Color contrast is intentional
- **WHEN** a generated artifact uses status colors, muted text, code tokens, highlighted lines, or diagram labels
- **THEN** foreground and background token pairs meet the project's report contrast threshold
- **AND** muted text, code comments, line numbers, and status labels remain legible in Codex and local browser rendering

#### Scenario: Code panels preserve layout
- **WHEN** a code or diff section contains long lines, highlighted lines, file paths, copy buttons, or line numbers
- **THEN** the code panel contains overflow within the panel instead of expanding the entire page width
- **AND** line numbers, highlighted lines, and token spans align without covering code text
- **AND** the panel remains usable in narrow and desktop viewports

#### Scenario: Mermaid panels preserve layout
- **WHEN** a Mermaid section renders an SVG diagram
- **THEN** the diagram is contained in a dedicated scrollable or responsive panel
- **AND** the SVG does not cover adjacent sections
- **AND** long labels do not cause the page body to overflow horizontally
- **AND** the Mermaid source fallback remains available below or near the rendered diagram

#### Scenario: Hover and focus states do not shift layout
- **WHEN** the user hovers or focuses cards, filters, tabs, copy buttons, navigation links, or evidence items
- **THEN** the visual state change does not move neighboring components enough to cause overlap, clipping, or reflow that hides content
- **AND** keyboard focus remains visible

### Requirement: Stress fixtures cover rich-report failure modes
The system SHALL include stress fixtures that exercise report content likely to break layout, rendering, syntax highlighting, navigation, or validation.

#### Scenario: Runtime stress fixture exists
- **WHEN** the `effective-interact` skill ships test fixtures
- **THEN** at least one runtime-cdn stress fixture includes long Chinese and English headings, many sections, long paths, multiple Mermaid diagram types, wide code, diff content, Markdown tables, nested JSON, evidence, verification, filters, tabs, and next actions

#### Scenario: Stress fixture is generated by the normal path
- **WHEN** the stress fixture is used in tests or manual validation
- **THEN** it is passed through the same `create-interaction.mjs` path used for normal interaction artifacts
- **AND** it is validated by `validate-interaction.mjs`

#### Scenario: Stress fixture proves rich content rendering
- **WHEN** browser validation runs against the runtime-cdn stress report
- **THEN** validation confirms that Markdown rendered, Mermaid diagrams attempted runtime rendering, code highlighting produced token markup, and source fallbacks are still present
- **AND** validation reports any runtime failures as explicit degraded or failed states

## MODIFIED Requirements

### Requirement: Interaction generator produces reliable HTML by default
The system SHALL provide a reusable generation path that converts structured JSON interaction input into a single `.html` artifact whose conclusions, evidence, source fallbacks, and runtime-rendered rich content are reliable in Codex and local browser environments.

#### Scenario: Generate implementation handoff report
- **WHEN** the agent provides structured artifact input with title, summary, sections, evidence, and verification state
- **THEN** the system writes one `.html` report under the configured report directory
- **AND** the report includes a conclusion-first summary, grouped navigation, rich content sections, evidence, verification status, and next actions
- **AND** the report declares its render mode in the HTML root or metadata

#### Scenario: Default output uses runtime-cdn rendering
- **WHEN** the report is generated without an explicit render mode
- **THEN** the system generates a `runtime-cdn` report optimized for viewing inside Codex and local browsers with network access
- **AND** the report includes pinned runtime dependencies for Markdown rendering, Markdown sanitization, Mermaid rendering, and code highlighting
- **AND** the report does not claim full offline rendering support unless an explicit offline-capable mode is selected

#### Scenario: Runtime-cdn report keeps critical content available
- **WHEN** a runtime-cdn report is opened before external libraries finish loading
- **THEN** the report title, summary, status, evidence, verification state, next actions, and source fallbacks remain readable
- **AND** rich content sections display pending, degraded, failed, or ready state instead of silently appearing complete

#### Scenario: Explicit pre-rendered mode remains available
- **WHEN** the artifact input explicitly selects `pre-rendered` mode
- **THEN** the system generates a report whose primary rich content does not require CDN scripts
- **AND** Markdown, Mermaid, and code sections either render before handoff or display explicit degraded status with source fallback
- **AND** the report remains subject to the same evidence, navigation, safety, and browser validation requirements where applicable

### Requirement: Runtime rendering is explicit and auditable
The system SHALL support browser runtime rendering through named render modes that declare dependencies, expose section-level render state, preserve source fallback, and fail visibly.

#### Scenario: Runtime mode declares dependencies
- **WHEN** runtime rendering is enabled for a report
- **THEN** the report declares pinned runtime library names, versions, loading URLs, and purposes in the HTML or visible runtime dependency panel
- **AND** the report includes source fallbacks for runtime-rendered Markdown, Mermaid, and code sections
- **AND** the report keeps primary conclusions and evidence readable if a runtime enhancement fails

#### Scenario: Runtime mode tracks section state
- **WHEN** a runtime-rendered section starts, succeeds, degrades, or fails
- **THEN** the report records that state on the affected section with a machine-readable attribute
- **AND** the report displays a visible user-facing status for that section
- **AND** one section failure does not prevent unrelated sections from rendering when their dependencies are available

#### Scenario: Runtime mode fails visibly
- **WHEN** a runtime library is unavailable, a diagram is invalid, Markdown sanitization is unavailable, or code highlighting fails
- **THEN** the affected section displays a visible fallback, degraded state, or error status
- **AND** the report preserves the source text needed to audit the content
- **AND** the page-level runtime status reflects that at least one section is degraded or failed

#### Scenario: Legacy runtime alias is handled deliberately
- **WHEN** existing artifact input uses the older `runtime` render mode value
- **THEN** the generator either maps it to `runtime-cdn` with a compatibility note or rejects it with a clear migration error
- **AND** tests define the chosen behavior

### Requirement: Rich content is rendered safely
The system SHALL render rich content through controlled runtime or pre-rendered paths without executing untrusted report content.

#### Scenario: Markdown is sanitized
- **WHEN** Markdown content is generated from user-controlled or mixed-trust input
- **THEN** the rendered HTML is sanitized before it is inserted into the report
- **AND** unsafe scripts, event handlers, unsupported protocols, and unsupported raw HTML are removed or escaped
- **AND** Markdown runtime rendering does not proceed for mixed-trust content if the sanitizer is unavailable

#### Scenario: Mermaid uses strict rendering
- **WHEN** Mermaid content is rendered at build time or runtime
- **THEN** the renderer uses `startOnLoad: false` and a strict security configuration where supported
- **AND** the report keeps the Mermaid source in an auditable fallback block
- **AND** a Mermaid render failure does not execute or expose unsafe diagram content as active HTML

#### Scenario: Code and paths are inert
- **WHEN** code snippets or file paths are included in a report
- **THEN** they are treated as inert text unless the report explicitly renders a safe local path reference
- **AND** the report does not execute code snippets or path-derived content
- **AND** copy controls copy text content rather than evaluating code or path-derived markup

#### Scenario: Runtime dependency declarations are not trusted content
- **WHEN** runtime library names, versions, URLs, or purposes are rendered in the report
- **THEN** those values are escaped or generated from trusted constants
- **AND** dependency metadata cannot inject script, event handlers, or unsupported protocols into the page

#### Scenario: Runtime and validation diagnostics are sanitized
- **WHEN** Mermaid pre-rendering, browser validation, or another report diagnostic path receives an error message containing local absolute paths, `file:///` URLs, token-shaped secrets, raw HTML, event handlers, or unsupported protocols
- **THEN** the interaction generator and validator replace sensitive local details with placeholders before writing HTML or JSON output
- **AND** the diagnostic remains short enough to explain the failure without exposing machine-specific or credential-like data

### Requirement: Interaction validation checks visual and interactive behavior
The system SHALL provide validation that checks generated artifacts for rendering, evidence, accessibility, runtime state, visual stability, responsive layout, and interaction readiness.

#### Scenario: Validate rich rendered content
- **WHEN** validation runs against a generated artifact
- **THEN** it verifies that Markdown tables/lists, Mermaid rendering targets, and code highlighting are present when those section types are used
- **OR** it reports that the section is runtime-dependent with a valid fallback, dependency declaration, and explicit pending/degraded/failed state

#### Scenario: Validate runtime-cdn completion
- **WHEN** validation runs against a runtime-cdn report in browser-required mode
- **THEN** it opens the report in a browser-capable environment
- **AND** it waits for runtime rendering to complete, degrade, fail, or time out
- **AND** it fails validation if required runtime sections remain silently pending without visible fallback or status

#### Scenario: Validate interaction controls
- **WHEN** validation runs against a generated artifact with filters, tabs, copy buttons, navigation, or focus effects
- **THEN** it verifies that the controls exist, have accessible labels or text, and produce the expected DOM state changes in a browser-capable environment
- **AND** interacting with controls does not hide primary conclusions, evidence, or verification state

#### Scenario: Validate responsive viewports
- **WHEN** validation runs in a browser-capable environment
- **THEN** it checks narrow, medium, and desktop viewport widths for blank output, missing primary content, body-level horizontal overflow, and major overlap of report controls and text
- **AND** failures identify the viewport and selector or region involved where possible

#### Scenario: Validate Mermaid visual containment
- **WHEN** validation runs against a report with Mermaid sections in a browser-capable environment
- **THEN** it verifies that each rendered Mermaid section has a non-empty rendered target or an explicit degraded/failed state
- **AND** it checks that rendered diagrams do not cover adjacent sections
- **AND** it checks for obvious SVG text or diagram content escaping the visible diagram container

#### Scenario: Validate code highlighting quality
- **WHEN** validation runs against a report with code sections in a browser-capable environment
- **THEN** it verifies that code blocks include the expected language class
- **AND** runtime-cdn reports produce highlight token markup when the highlighter succeeds
- **AND** line numbers and highlighted lines do not cover code text

#### Scenario: Browser-assisted validation can degrade explicitly
- **WHEN** browser automation or local Chrome is unavailable and browser-required mode is not requested
- **THEN** validation reports the degraded coverage explicitly
- **AND** validation does not silently claim browser-only rendering checks passed

#### Scenario: Browser-required validation fails without browser coverage
- **WHEN** browser automation or local Chrome is unavailable and browser-required mode is requested
- **THEN** validation fails with a clear reason
- **AND** the result explains that runtime-cdn visual quality could not be verified
